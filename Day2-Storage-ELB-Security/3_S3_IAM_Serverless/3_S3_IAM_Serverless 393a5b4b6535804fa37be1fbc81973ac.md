# 요약

## 1. 🔑 S3 스토리지 활용 및 고급 기능 개요

- **S3 Glacier (빙하, 냉동실):** 자주 접근하지 않는 장기 보관용 데이터(로그, 백업 파일)를 저렴한 비용으로 저장하는 콜드 스토리지 클래스입니다.
- **버전 관리 (Versioning):** 객체의 수정/삭제 이력을 모두 보존하여 실수로 파일을 덮어쓰거나 지웠을 때 과거 상태로 완벽히 복구할 수 있는 양날의 검(안전성 확보 vs 용량 비용 증가) 기능입니다.

---

## 🛠️ 2. S3 정적 웹 사이트 호스팅 트러블슈팅

### 🚨 발단: `404 NoSuchKey (index.html)` 및 CSS 깨짐 현상

S3 호스팅 엔드포인트 주소로 접속했으나 사이트가 깨지거나 대문 페이지를 찾지 못하는 오류가 발생했습니다.

### 🔍 원인 및 해결 프로세스

1. **버킷 정책 및 JSON 문법 수정:** 초기 버킷 정책 작성 시 여러 문장이 중복되거나 객체 레벨(`/*`) 경로가 누락되어 `s3:GetObject` 액션이 정상 작동하지 않는 문법 에러가 있었습니다. `AWS Policy Generator` 설정을 기반으로 정책 구조를 하나로 병합했습니다. (`bucket_policy.json` 참고)
2. **경로 불일치 해결:** 내 컴퓨터에 있던 프로젝트 폴더 자체를 S3에 통째로 드래그하여 올리다 보니, `index.html`이 최상위 루트가 아닌 하위 폴더에 숨겨져 엔드포인트가 인식하지 못했습니다. 또한 내부의 `css/`, `js/`, `assets/` 디자인 자원들의 상대 경로가 깨져서 UI가 무너진 것이었습니다.
3. **최종 조치:** 기존 폴더 구조를 해제하고, `index.html`과 디자인 하위 폴더들이 **S3 버킷 최상위 경로에 나란히 수평적으로 배치되도록 재업로드**하여 사이트를 완전히 정상화시켰습니다.

---

## 🔒 3. 클라우드 보안 및 IAM 대원칙

- **루트 계정(Root Account) 사용 금지:** 모든 자원을 삭제할 수 있는 강력한 권한을 가진 루트 계정은 가입 직후 MFA를 설정하고 일상 업무에서 완전히 배제합니다.
- **최소 권한의 원칙 (Principle of Least Privilege):** 각 유저나 서비스에는 업무 및 실습에 필요한 딱 최소한의 권한만을 정책(`Policy`)으로 정의하여 할당해야 보안 사고를 예방할 수 있습니다.
- **역할(Role)의 활용:** 사람이 아닌 EC2 인스턴스나 Lambda 함수가 다른 AWS 자원(S3 등)에 임시로 안전하게 접근할 수 있도록 자격 증명을 위임하는 용도로 역할을 사용합니다.

## ⚡ 4. 서버리스(Serverless) 컴퓨팅: AWS Lambda

- **개념:** 개발자가 서버의 OS나 인프라 관리에 전혀 신경 쓰지 않고, 오직 백엔드 실행 **코드만 작성해서 올려두면 작동**하는 컴퓨팅 서비스입니다.
- **이벤트 기반 작동 구조:** 24시간 켜져 있는 EC2 서버와 달리 평소에는 요금이 나가지 않다가, **"S3에 새로운 사진이 업로드됨"**과 같은 특정 트리거(이벤트)가 발생할 때만 밀리초(ms) 단위로 슥 깨어나서 코드를 딱 실행하고 다시 잠드는 비용 최적화 아키텍처입니다.
- **실무 시나리오:** 사용자가 고용량 프로필 이미지를 S3에 업로드하면 ➡️ S3가 람다를 트리거 ➡️ 람다가 순식간에 모바일용 섬네일 이미지로 리사이징 후 다시 S3에 저장하고 종료되는 파이프라인의 효율성을 학습했습니다.

## 1. 기존 리소스 삭제 및 의존성 이해

AWS에서 자원을 삭제할 때는 리소스 간의 **의존성(Dependency)** 순서를 지켜야 오류 없이 깔끔하게 삭제됩니다.

### 짚고 넘어가기: 리소스 삭제 트러블슈팅

- **보안 그룹(SG) 삭제 실패 원인:** 특정 보안 그룹(예: `Myalbsg`)을 다른 자원(서버, 로드 밸런서 등)이 사용 중이거나, Auto Scaling 그룹이 아직 삭제 중(Terminating)일 경우 즉시 삭제되지 않습니다.
- **올바른 삭제 순서 프로세스:**
    1. **인바운드 규칙 수정:** 보안 그룹 체이닝을 해제하고 단독 규칙(HTTP Anywhere)으로 변경 및 저장합니다.
    2. **Auto Scaling 그룹 삭제:** (가장 먼저 시작해야 하나 완료까지 시간이 다소 소요됨)
    3. **로드 밸런서(ALB) 삭제:** ASG 삭제 중이라도 로드 밸런서는 먼저 삭제가 가능합니다.
    4. **대상 그룹(Target Group) 삭제:** 로드 밸런서가 삭제되면 대상 그룹도 삭제할 수 있게 됩니다.
    5. **EC2 인스턴스 종료 확인:** 인스턴스가 최종 '종료됨(Terminated)' 상태인지 확인합니다.
    6. **시작 템플릿 및 AMI 삭제:** 비용 방지를 위해 AMI 등록 취소 시 **연결된 스냅샷 삭제 옵션**을 반드시 함께 체크합니다.
    7. **보안 그룹 삭제:** 의존성이 모두 사라진 후 최종적으로 보안 그룹을 삭제합니다.

## 2. Amazon S3 스토리지 실습

S3(Simple Storage Service)는 전 세계적인 규모의 **객체(Object) 스토리지**입니다. 디렉터리나 폴더 구조를 가진 일반 디스크와 달리, 분산 저장되는 논리적인 공간입니다.

### 2.1 버킷(Bucket) 생성 및 개념

- **버킷:** 데이터가 담기는 최상위 논리적 컨테이너 (별도의 용량 제한이 없으며 사용한 만큼 비용 지불).
- **버킷 이름 규칙:** 기존에는 전 세계에서 유일해야 했으나, 최근 계정/리전 기준 네임스페이스 옵션이 추가되었습니다. 단, 중복을 피하기 위해 `이름이니셜-날짜` 조합 등을 활용하는 것이 좋습니다.
- **옵션 설정:** 실습을 위해 **ACL(액세스 제어 목록) 활성화** 및 객체 소유권을 객체 작성자(Bucket owner preferred)로 설정합니다.

### 2.2 퍼블릭 액세스(Public Access) 권한 부여

외부에서 URL을 통해 S3에 업로드된 객체(예: 이미지, PDF)에 접근하게 하려면 **두 가지 권한**이 모두 열려 있어야 합니다.

1. **버킷(Bucket) 수준:** 버킷 권한 탭 $\rightarrow$ '모든 퍼블릭 액세스 차단' 해제
2. **객체(Object) 수준:** 업로드된 파일 선택 $\rightarrow$ 작업 $\rightarrow$ '퍼블릭으로 설정' 클릭

> ⚠️ **주의:** 버킷을 퍼블릭으로 설정했다고 해서, 이후 새롭게 업로드되는 객체들이 자동으로 퍼블릭 상태가 되지는 않습니다. (수동 설정 필요)
> 

### 2.3 버킷 정책(Bucket Policy)을 통한 자동 퍼블릭 설정

새로운 파일이 올라올 때마다 자동으로 퍼블릭 권한을 부여하려면 **JSON 형태의 버킷 정책**을 적용해야 합니다. AWS Console에서 직접 작성하기 어려우므로 **AWS Policy Generator** 도구를 활용합니다.

#### 💡 AWS Policy Generator 설정 값

- **Select Type of Policy:** `S3 Bucket Policy`
- **Effect:** `Allow`
- **Principal:**  (모든 사람)
- **Actions:** `GetObject`, `PutObject` 체크
- **Amazon Resource Name (ARN):** S3 버킷 권한 탭의 **버킷 ARN** 복사 후 붙여넣기

#### 🛠️ 제이슨(JSON) 정책 수정 필수 팁

생성된 JSON 코드를 S3 정책 편집기에 붙여넣을 때, 리소스 끝단에 `/*`**를 반드시 수동으로 붙여주어야 버킷 하위의 모든 객체에 정책이 적용됩니다

```
"Resource": "arn:aws:s3:::양동이-이름/*"
```

## 3. S3 고급 기능 활용

### 3.1 스토리지 클래스 전환 (비용 절감)

S3는 데이터의 접근 빈도에 따라 비용을 최적화할 수 있는 다양한 스토리지 클래스를 제공합니다.

- **Standard (기본):** 자주 접근하는 데이터용.
- **Glacier Flexible / Deep Archive (냉동실):** 접근 빈도가 매우 낮은 장기 보관용 데이터. 비용이 매우 저렴합니다.
- **데이터 복원(해동):** Glacier에 있는 파일은 즉시 읽을 수 없으며, **복원(Restoration)** 신청 후 일정 시간(표준 3~5시간, 대량 최대 48시간)을 기다려야 임시로 다시 읽을 수 있습니다.

### 3.2 버전 관리(Versioning)의 양날의 검

- 파일의 수정/삭제 이력을 모두 기록하여 데이터를 보호할 수 있습니다.
- **비용 폭탄 주의:** 버전 관리를 활성화하면 눈에 보이지 않는 이전 버전 파일들이 계속 쌓이므로, 용량이 급증하여 비용 이슈가 발생할 수 있습니다. **반드시 지워지면 안 되는 핵심 경로에만 제한적으로 활성화**해야 합니다.

### 3.3 정적 웹사이트 호스팅 (Static Website Hosting)

S3 버킷을 활용하면 별도의 웹 서버(EC2 등)를 구축하지 않고도 웹사이트를 배포할 수 있습니다.

1. 속성 탭 맨 아래 **정적 웹사이트 호스팅 활성화** $\rightarrow$ 인덱스 문서에 `index.html` 입력.
2. 배포할 웹 소스코드 파일들을 버킷 최상위에 업로드 (※ 폴더째 올리지 말고 파일들을 전체 선택하여 업로드).
3. 속성 탭 하단에 생성된 **버킷 웹사이트 엔드포인트 URL**로 접속 시 정상 구동 확인.

> 🌐 **서버리스(Serverless) 아키텍처 확장**
> 
> 
> S3(정적 웹사이트) + AWS Lambda(서버리스 코드 실행) + DynamoDB(분산 NoSQL DB)를 조합하면, 단 한 대의 가상 서버 없이도 완벽한 고성능 웹 서비스를 최소한의 비용으로 운영할 수 있습니다.
> 

## 4. 클라우드 보안 및 IAM (Identity and Access Management)

### 4.1 루트 계정(Root Account) 사용 금지

- 가입 시 사용한 이메일 계정(루트 계정)은 결제 정보 및 모든 권한을 가진 주인이므로, 평소에 절대 직접 사용하거나 직원들과 공유해서는 안 됩니다.
- 대신 **IAM**을 통해 개별 직원용 계정을 생성하고 필요한 권한만 할당해야 합니다.

### 4.2 IAM의 핵심 구성 요소

- **사용자(User):** 실제 로그인을 하여 작업하는 사람(직원).
- **정책(Policy):** 무엇을 할 수 있고 없는지 정의한 권한 목록(JSON 형태). 읽기 전용(`ReadOnly`), 전체 권한(`FullAccess`) 등이 있습니다.
- **역할(Role):** 사람이 아닌 AWS 자원(예: EC2 인스턴스)에게 권한을 부여할 때 사용합니다. (예: EC2 웹서버가 S3에서 이미지를 긁어와야 할 때 EC2에 S3 접근 '역할'을 부여)

### 4.3 보안의 대원칙: 최소 권한의 원칙 (Principle of Least Privilege)

> 🚨 **보안 사고의 주원인:** 대부분의 클라우드 보안 사고는 해킹 기술의 문제보다 **"관리 편의를 위해 직원이나 자원에 권한을 너무 과도하게 열어두어서"** 발생합니다.
> 
- 귀찮더라도 각 사용자에게 업무에 필요한 **최소한의 권한**만 타이트하게 부여하는 습관이 클라우드 보안 및 비용 통제의 핵심입니다.

## 🚀 람다가 등장한 배경: 기존 서버와의 차이점

기존의 일반적인 백엔드 개발(예: EC2 가상 머신 이용)은 아래와 같은 과정이 필요했습니다.

1. 컴퓨터(서버)를 대여한다.
2. OS를 설치하고 Java나 Node.js 같은 환경을 세팅한다.
3. 24시간 내내 서버를 켜두고 손님(요청)을 기다린다. (손님이 없어도 돈이 계속 나감 💸)

반면 람다(Lambda)는 이 모든 귀찮은 과정을 생략합니다.

- **서버 컴퓨터를 관리할 필요가 전혀 없습니다**. AWS가 알아서 관리해 줍니다.
- 24시간 켜져 있는 게 아니라, **"특정 이벤트가 발생했을 때만"** 찰나의 순간 컴퓨터가 켜져서 내 코드를 실행하고 다시 꺼집니다.

## ⚡ 람다는 어떻게 작동하나요? (이벤트 기반 작동)

람다는 혼자서 작동하지 않고 보통 **'트리거(Trigger)'** 역할을 하는 다른 서비스와 연결되어 움직입니다. 방금 실습하신 S3와 연결하는 시나리오가 가장 대표적인 백엔드 실무 예시입니다.

### 📸 실무 예시: "이미지 섬네일 자동 생성기"

1. 사용자가 내 웹사이트(`ljh.com`)에 프로필 사진(고화질 10MB짜리)을 올립니다.
2. 사진이 S3 버킷에 저장(이벤트 발생)됩니다.
3. S3가 람다에게 톡을 보냅니다. *"방금 사진 하나 새로 들어왔어!"*
4. 잠들어 있던 **람다 함수가 0.1초 만에 깨어나서** 10MB짜리 사진을 백엔드 코드(Java 등)로 100KB짜리 작은 섬네일로 리사이징합니다.
5. 람다가 섬네일 이미지를 다시 S3에 저장하고 **스스로 종료**됩니다.

## 🔥 백엔드 개발자가 람다를 쓰면 좋은 점 (장점)

### 1. 비용이 파격적으로 절약됩니다.

- 24시간 켜놓는 서버는 사용자가 없어도 매달 고정 비용이 나갑니다.
- 람다는 코드가 실제로 실행된 시간(밀리초 단위, ms)과 **호출 횟수**만큼만 돈을 냅니다. 만약 한 달 동안 아무도 요청을 안 보냈다면 요금은 **0원**입니다. 심지어 매달 100만 번의 실행까지는 무료 제공량이 주어집니다.

### 2. 인프라 확장(Scaling) 고민이 사라집니다.

- 갑자기 수만 명의 사용자가 동시에 접속해서 이미지를 올리면 일반 서버는 터져버립니다.
- 람다는 요청이 1개 들어오면 1개, 1,000개 들어오면 순식간에 람다 컴퓨터 1,000개를 동시에 개별적으로 띄워서 처리합니다. 개발자가 트래픽 처리를 위해 서버를 증설할 필요가 없습니다.

## ⚠️ 람다를 쓸 때 주의할 점 (단점)

- **콜드 스타트(Cold Start):** 람다는 평소에 꺼져 있다가 요청이 오면 그제야 컴퓨터를 띄우고 코드를 로딩합니다. 그래서 **오랜만에 첫 요청**이 들어올 때는 **첫 실행이 약 0.5초~몇 초간 살짝 느려지는 현상**이 있습니다. (특히 자바 계열이 이 콜드 스타트 시간이 조금 있는 편입니다.)
- **최대 실행 시간 한계:** 람다는 한 번 켜지면 최대 **15분** 동안만 딥러닝 연산이나 대용량 데이터 분석처럼 **15분이 넘어가는 무거운 작업에는 적합하지 않습니다.**

# 🔐 AWS IAM (Identity, Access Management)

> 💡 **한 줄 요약:** AWS의 모든 자원(EC2, S3 등)에 **"누가(Identity)"**, **"어떤 권한으로(Access)"** 접근할 수 있는지를 안전하게 제어하는 **인증 및 권한 관리 시스템**입니다.
> 

## 1. 👥 IAM의 4가지 핵심 구성 요소

IAM을 이해하려면 아래 4가지 개념만 확실히 알면 됩니다.

### ① 사용자 (User)

- **실제 AWS를 사용하는 '사람' 또는 '애플리케이션'** 한 명 한 명을 의미합니다.
- 예: 개발자 `이지현` 계정, 인턴 계정 등.
- 각 사용자마다 고유한 로그인 비밀번호나 API Access Key를 가집니다.

### ② 그룹 (Group)

- 사용자들의 권한 관리를 편하게 하기 위해 묶어둔 '방'입니다.
- 예: `Developer 그룹`(개발 권한), `Admin 그룹`(모든 권한).
- 그룹에 권한을 부여하면, 그 그룹에 속한 유저들은 자동으로 똑같은 권한을 갖게 됩니다. (유저마다 일일이 권한 설정을 안 해도 돼서 관리가 편해집니다.)

### ③ 정책 (Policy)

- **"무엇을 할 수 있고, 할 수 없는지"** 명시해 둔 권한 설명서(JSON 파일)입니다.
- AWS에서 이미 만들어둔 정책(예: `AmazonEC2FullAccess`)을 쓸 수도 있고, 내가 직접 만들 수도 있습니다.
- **유저, 그룹, 역할**에 이 **'정책**'을 스티커처럼 붙여서 권한을 부여합니다.

### ④ 역할 (Role)

- 사람이 아니라 **'EC2 서버'나 '람다 함수' 같은 AWS 서비스나 자원에게 임시로 권한을 줄 때** 사용합니다.
- 💡 **예시:** 아까 실습할 때 EC2 웹 서버가 S3 스토리지에 있는 이미지 파일을 읽어와야 한다면, EC2 서버에 `S3ReadOnly`라는 역할(Role)을 부여해서 안전하게 접근할 수 있도록 만듭니다.

## 2. 🛡️ 보안을 위한 실무 필수 원칙 (Best Practice)

### ❌ 루트(Root) 계정은 일상 업무에 쓰지 않기

- 처음 AWS 가입할 때 만든 이메일 계정이 '루트 계정'입니다. 이 계정은 모든 것을 삭제할 수 있는 초강력 권한을 가집니다.
- **실무 규칙:** 루트 계정은 가입 직후 MFA(2차 인증)를 걸어두고 깊은 곳에 봉인합니다. 평소 실습이나 업무를 할 때는 **최소 권한을 가진 IAM 유저를 따로 생성해서 사용**해야 보안 사고(비용 폭탄 등)를 막을 수 있습니다.

### 📉 최소 권한 부여 원칙 (Least Privilege)

- 개발자에게 굳이 결제(Billing) 권한이나 인프라 전체 삭제 권한을 줄 필요가 없습니다.
- 딱 **업무에 필요한 만큼의 최소 권한만 정책으로 허용**하는 것이 기본입니다.

- **사람이 아니라 서비스(EC2 등)에 임시 권한을 줄때는?** ➡️ **역할(Role)**을 부여
