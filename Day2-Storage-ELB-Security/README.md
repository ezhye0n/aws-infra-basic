# 🚀 Day 2: 고가용성 인프라 구축, S3 호스팅 및 클라우드 보안/서버리스

2일차에는 서비스의 중단 없는 운영을 위한 **고가용성(High Availability)** 아키텍처를 설계하고, **S3 정적 웹 호스팅** 과정에서의 트러블슈팅, 그리고 클라우드 보안의 핵심인 **IAM**과 **서버리스(Lambda)**의 기본 개념을 연계하여 학습하고 실습했습니다.

---

## 📂 디렉토리 구조 및 실습 내용

### 1. [1_EC2_AMI_HighAvailability](./1_EC2_AMI_HighAvailability/)
- **핵심 개념:** 온디맨드(On-Demand) 컴퓨팅과 가상화 하이퍼바이저(Hypervisor)의 이해
- **실습 내용:** - 기본 Nginx 웹 서버(1번)를 구축한 후, 이를 기반으로 **AMI(Amazon Machine Image)**를 생성
  - 생성된 AMI 스냅샷을 활용하여 동일한 설정의 2번 웹 서버를 신속하게 복제 및 이중화 환경 설계

### 2. [2_ALB_AutoScaling_CloudWatch](./2_ALB_AutoScaling_CloudWatch/)
- **핵심 개념:** 로드 밸런싱(부하 분산), 인프라 탄력성(Elasticity) 및 수평적 확장(Scale-out)
- **실습 내용:**
  - **보안 그룹 체인(체인 방화벽):** 외부 트래픽이 무조건 로드 밸런서를 거쳐서만 웹 서버에 도달하도록 인바운드 보안 아키텍처 설계
  - **동적 크기 조정 정책:** CPU 사용률 임계값(50%) 기준 및 워밍업(60초) 설정을 반영한 ASG(Auto Scaling Group) 구성
  - **부하 테스트 검증:** 리눅스 터미널 환경에서 `stress` 툴(8코어 부하)을 이용해 CloudWatch 경보 발생 및 인스턴스가 2대에서 **최대 용량인 5대까지 동적 확장**되는 매커니즘 확인

### 3. [3_S3_IAM_Serverless](./3_S3_IAM_Serverless/)
- **핵심 개념:** 객체 스토리지 수명 주기, 최소 권한 원칙(Least Privilege), 이벤트 기반 서버리스
- **실습 내용:**
  - **S3 정적 웹 사이트 호스팅:** 부트스트랩 테마를 활용한 호스팅 및 버킷 정책 JSON 제어
  - **클라우드 보안 (IAM):** 루트 계정 사용 금지 원칙 준수, User/Group/Policy/Role의 차이점 및 EC2-S3 간 임시 권한(Role) 위임 학습
  - **서버리스 (Lambda):** S3 업로드 이벤트 등을 트리거로 작동하는 비용 효율적 컴퓨팅 서비스의 작동 원리 및 이미지 썸네일 자동화 아키텍처 이해

## 🛠️ 핵심 트러블슈팅 기록 (Troubleshooting)

### 📌 S3 호스팅 에러: `404 NoSuchKey (index.html)` 및 CSS 깨짐 현상
- **문제 상황:** S3 호스팅 엔드포인트 접속 시 대문 페이지를 찾지 못하는 404 에러가 발생하거나, 텍스트만 나열되고 부트스트랩 테마(CSS/Assets)가 완전히 깨져서 출력됨.
- **원인 분석:** - S3 버킷 최상위 경로(Root, `/`)가 아닌 하위 특정 폴더 내부에 `index.html`이 배치되어 호스팅 엔진이 진입점을 인식하지 못함.
  * 소스 코드 내부의 상대 경로(`<link href="css/styles.css">`)와 달리, S3 버킷 내에 스타일시트 및 미디어 폴더(`css/`, `assets/`)가 나란히 업로드되지 않아 리소스 로드 실패(`403 Forbidden` / `Failed to load resource`).
- **해결 방법:** - 로컬 프로젝트 폴더 '내부'로 진입하여 `index.html`과 하위 리소스 폴더들을 **S3 버킷의 최상위 경로에 동일한 층위로 재배치**하여 드래그 앤 드롭 업로드 수행.
  - 버킷 정책(`bucket_policy.json`)의 와일드카드(`/*`) 규칙이 하위 리소스까지 정상 적용됨을 확인하고 강력 새로고침(`Ctrl + F5`)을 통해 사이트 정상화 완료.

## 💡 학습 회고
- **아키텍처적 보안의 중요성:** 단순히 포트를 여는 것을 넘어, ALB의 보안 그룹만을 신뢰하도록 EC2 보안 그룹을 체이닝하는 방식을 통해 백엔드 인프라의 망 격리와 보안 강화 원칙을 체득했습니다.
- **탄력성(Elasticity) 체감:** 서버가 터지는 상황을 방치하는 것이 아니라, CloudWatch 지표와 오토 스케일링을 연동하여 시스템이 스스로 수평 확장(Scale-out)하는 과정을 검증하며 클라우드 기반 아키텍처의 강력함을 경험했습니다.
