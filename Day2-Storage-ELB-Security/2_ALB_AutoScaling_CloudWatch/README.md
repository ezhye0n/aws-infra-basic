## 🚀 ALB 로드 밸런서 구축 및 Auto Scaling 동적 크기 조정 실습

# 요약

## 1. 🔑 핵심 개념 및 설계 구조

- **ALB (Application Load Balancer):** OSI 7계층(애플리케이션 계층)에서 작동하는 부하 분산기로, URL 경로, HTTP 헤더 등을 기반으로 트래픽을 여러 타겟 그룹으로 똑똑하게 나누어 줍니다.
- **Auto Scaling Group (ASG):** 애플리케이션의 부하 상태(CPU 사용률 등)에 따라 EC2 인스턴스를 자동으로 늘리거나(`Scale-out`) 줄여서(`Scale-in`) 서비스의 고가용성과 비용 최적화를 동시에 달성합니다.

---

## 🛡️ 2. 주요 실습 및 방화벽 설계 체인

### 🔒 보안 그룹 체인 (체인 방화벽) 설정

- 웹 서버가 외부에 직접 노출되면 DDoS 공격이나 포트 스캔에 취약해집니다.
- **해결 체인:** EC2 인스턴스의 보안 그룹 인바운드 규칙에서 `0.0.0.0/0`을 열지 않고, **"ALB가 사용하는 보안 그룹"** 자체를 소스(Source)로 지정했습니다.
- **결과:** 외부 사용자는 무조건 ALB(로드 밸런서)를 거쳐서만 웹 서버에 도달할 수 있으며, 인스턴스로 직접 치고 들어오는 악성 트래픽을 원천 차단하는 방화벽 체인을 구축했습니다.

---

## ⚡ 3. 부하 모니터링 및 Stress 테스트 검증

CloudWatch 경보와 오토 스케일링 정책을 연동하여 시스템이 트래픽 폭주에 대응하는 과정을 시뮬레이션했습니다.

### 1단계: CloudWatch 경보 생성

- 대상 추적 크기 조정 정책을 통해 **"평균 CPU 사용률이 50%를 초과"**하면 알림을 발생시키고 인스턴스를 증설하도록 임계값을 모니터링 대시보드에 연동했습니다.
- 부하 감지 시 경보가 작동하며 **SNS(Simple Notification Service)**를 통해 지정된 관리자 이메일로 구독 알림이 발송되는 흐름을 세팅했습니다.

### 2단계: 리눅스 터미널 부하 발생 (Stress Test)

- 웹 기반 터미널을 통해 1, 2번 서버에 동시에 접속한 뒤, 아래 명령어를 주입하여 인위적으로 CPU에 대규모 병목을 발생시켰습니다.
    
    ```bash
    # 8개의 코어를 풀 가동하여 CPU 사용률을 800%로 끌어올리는 부하 명령어
    stress --cpu 8 --timeout 600
    ```
    

### 3단계: 실시간 인스턴스 자동 확장(Scale-out) 모니터링

- **결과:** 부하 주입 직후 CloudWatch 그래프의 CPU 사용률 곡선이 급상승하며 경보(`In alarm`) 상태로 전환되었습니다.
- 오토 스케일링 그룹이 이를 감지하여 미리 정의한 사양대로 인스턴스를 **최대 5대까지 순차적으로 띄우며(Scale-out)** 트래픽을 분산 수용하는 클라우드의 탄력성 메커니즘을 최종 확인했습니다.

# 🚀 [실습] ALB 구축 및 CloudWatch 부하 모니터링

## 1. 💡 핵심 개념 요약

- **Application Load Balancer (ALB):** OSI 7계층(Application)에서 동작하는 로드 밸런서. HTTP/HTTPS 프로토콜에 특화되어 트래픽을 분산함.
- **보안 그룹 체인 (Security Group Chaining):** 웹 서버가 아무나의 접속을 받지 않고, **오직 로드 밸런서(ALB)를 통과한 트래픽만 수신**하도록 방화벽끼리 엮어 보안을 강화하는 실무 필수 패턴.
- **Amazon SNS (Simple Notification Service):** 관리자가 24시간 콘솔을 안 봐도 서버 경보(CPU 과부하 등) 발생 시 이메일, 푸시 등으로 알려주는 알림 서비스.
- **CloudWatch 경보(Alarm):** CPU 사용률 등 지표(Metric)가 임계값(예: 50% 이상)을 넘으면 SNS와 연동해 경보를 발생시킴.

## 2. 🔀 실습 1: ALB(애플리케이션 로드 밸런서) 생성 및 연동

### 1단계: ALB 생성 및 네트워크 매핑

1. `EC2 콘솔` ➡️ 왼쪽 메뉴 `로드 밸런싱` ➡️ `로드 밸런서` ➡️ **[로드 밸런서 생성]** 클릭.
2. **Application Load Balancer**의 **[생성]** 버튼 클릭.
3. **기본 구성:** * 이름: `my-alb`
    - 체계: `인터넷 경계 (Internet-facing)`
    - IP 주소 유형: `IPv4`
4. **네트워크 매핑:**
    - VPC: `my-VPC` 선택
    - 매핑(가용 영역): **2개 이상의 가용 영역(AZ) 서브넷을 모두 체크** (단일 존은 생성 불가 ❌).

### 2단계: 로드 밸런서 전용 보안 그룹(SG) 생성

1. 보안 그룹 항목 옆의 **[새 보안 그룹 생성 (Create security group)]** 링크 클릭 (새 탭 열림).
2. **보안 그룹 구성:**
    - 이름 및 설명: `my-alb-SG`
    - VPC: `my-VPC`
3. **인바운드 규칙 추가:** `HTTP` | `80` | `Anywhere-IPv4 (0.0.0.0/0)` ➡️ **[보안 그룹 생성]**
4. 기존 로드 밸런서 생성 창으로 돌아와 새로고침 후 `my-alb-SG` 선택 (**기본 디폴트 SG는 해제**).

### 3단계: 대상 그룹(Target Group) 생성 및 리스너 연결

1. 리스너 및 라우팅 항목에서 **[대상 그룹 생성 (Create target group)]** 링크 클릭 (새 탭 열림).
2. **대상 그룹 설정:**
    - 대상 유형 선택: `인스턴스 (Instances)`
    - 대상 그룹 이름: `web-server-TG`
    - 프로토콜/포트: `HTTP` | `80`
    - VPC: `my-VPC` ➡️ **[다음]** 클릭.
3. **대상 등록:** 만들어둔 1번 서버(`my-web01`), 2번 서버(`my-web02`)를 모두 체크 ➡️ **[아래에 보류 중인 것으로 포함]** 클릭 ➡️ **[대상 그룹 생성]**.
4. 기존 로드 밸런서 생성 창으로 돌아와 새로고침 후 앤드포인트 작업에 `web-server-TG` 연결 ➡️ **[로드 밸런서 생성]** 클릭.

## 3. 🔒 실습 2: 보안 그룹 체인 (체인 방화벽) 설정

> ⚠️ **중요:** 웹 서버가 개별 퍼블릭 IP로 직접 들어오는 접근은 차단하고, 로드 밸런서 주소로 오는 정상 트래픽만 받도록 설정합니다.
> 
1. `EC2 콘솔` ➡️ `보안 그룹` ➡️ 어제 만든 웹 서버용 보안 그룹(`my-web-SG`) 선택.
2. **[인바운드 규칙 편집]** 클릭.
3. 기존에 열어두었던 **일반 HTTP (0.0.0.0/0) 규칙을 삭제(X)** 함.
4. **새 규칙 추가:** * 유형: `HTTP` (80포트)
    - 소스: `사용자 지정` ➡️ 검색창 클릭 후 **`my-alb-SG` (방금 만든 ALB 보안 그룹)** 선택.
5. **[규칙 저장]** 클릭.

> 🌐 **확인:** 이제 개별 서버 퍼블릭 IP 주소로는 브라우저 접속이 차단되지만, **ALB의 DNS 이름**으로 접속하면 1번 서버와 2번 서버가 새로고침할 때마다 번갈아가며 정상 노출(로드 밸런싱)됩니다.
> 

## 4. 🚨 실습 3: SNS 알림 & CloudWatch CPU 부하 테스트

### 1단계: SNS(알림 서비스) 주제 및 구독 생성

1. AWS 콘솔에서 `Simple Notification Service (SNS)` 검색 후 이동.
2. **주제(Topic) 생성:** `주제 생성` ➡️ 유형: **[표준]** ➡️ 이름/표시이름: `admin-topic` 입력 후 생성.
3. **구독(Subscription) 생성:** 방금 만든 주제 내부에서 `구독 생성` 클릭.
    - 프로토콜: `이메일 (Email)`
    - 엔드포인트: **본인의 실제 이메일 주소** 입력 후 생성.
4. 📬 **이메일 인증 필수:** 본인 이메일 수신함 확인 ➡️ AWS에서 온 메일 안의 **[Confirm Subscription]** 링크를 클릭하여 상태를 `확인됨(Confirmed)`으로 변경.

### 2단계: CloudWatch 경보 생성

1. `EC2 콘솔` ➡️ `인스턴스` ➡️ 1번 웹 서버(`my-web01`) 체크.
2. `작업` ➡️ `모니터링 및 문제 해결` ➡️ **[CloudWatch 경보 관리]** 클릭.
3. **경보 설정:**
    - 알림 송신 대상: `admin-topic` 선택.
    - 경보 임계값 조건: **[CPU 사용률]** 기준, **`50% 이상`** 상태가 **`5분`** 지속될 때 경보 발령 설정 후 생성.

### 3단계: 리눅스 터미널 부하 발생 (Stress 테스트)

MobaXterm을 통해 **1번 웹 서버(`my-web01`)** 터미널에 접속하여 아래 명령어를 차례대로 입력합니다.

```
# 1. 관리자 권한 전환
sudo -i

# 2. stress 도구 설치 상태 확인
stress --version

# 3. CPU 400% 부하를 10분(600초) 동안 지속하는 명령 실행 (공회전 부하)
stress --cpu 400 --timeout 600
```

## 5. 🔍 확인 및 다음 과제 (오후 세션)

- 현재 `stress` 명령어로 인해 1번 서버의 CPU 사용률이 폭발적으로 상승한 상태입니다.
- **5분 뒤 결과:** CloudWatch 경보 상태가 빨간색(`경보 상태`)으로 변경되며, 등록한 이메일로 "서버 위험 알림 메일"이 수신되는 것을 확인하고 휴식을 취합니다.
- 🛠️ **다음 차시 예고:** 지금은 알림만 오고 서버가 늘어나지 못하므로, 다음 시간에는 이 부하 지표와 연동되어 서버 대수가 스스로 늘어나는 **오토 스케일링(Auto Scaling)** 설정을 진행합니다.

## 1. 💡 핵심 개념 요약

- **탄력성 (Elasticity):** 사람이 개입하지 않아도 주말, 심야, 출퇴근 시간 등 트래픽 변동에 따라 인프라가 스스로 늘어났다(`Scale-out`) 줄어드는(`Scale-in`) 클라우드의 핵심 성질.
- **수평적 조정 (Horizontal Scaling / Scale-out):** 부족한 부하를 채우기 위해 서버의 체급을 올리는 것이 아니라, **동일한 사양의 서버 개수(수량)를 옆으로 늘려가며** 부하를 분산하는 방식 (실습에서 진행 중인 방식).
- **EC2 인스턴스 연결 (Instance Connect):** MobaXterm 같은 외부 툴 없이도 AWS 웹 콘솔 환경에서 직접 브라우저 기반 터미널로 서버에 급히 접속할 수 있는 기능.

![image.png](2_ALB_AutoScaling_CloudWatch/image%201.png)

## 2. 🔥 실습 1: 웹 기반 터미널 접속 및 동시 부하(Stress) 테스트

> ⚠️ **주의:** 수동으로 만들었던 이전 웹 서버 2대는 ASG의 관리 대상이 아닙니다. 새로 생성된 `web-asg` 태그가 붙은 **새 인스턴스 2대**에 각각 접속해야 합니다.
> 

### 1단계: 웹 터미널로 두 서버 동시 접속

1. `EC2 콘솔` ➡️ `인스턴스` 이동.
2. 이름이 `web-asg`로 시작하는 첫 번째 인스턴스를 체크하고 상단의 **[연결 (Connect)]** 클릭.
3. 기본값 그대로 두고 하단의 [연결]을 누르면 브라우저에 새 탭으로 터미널이 열림.
4. **두 번째 `web-asg` 인스턴스**도 동일하게 체크하여 또 다른 새 탭으로 터미널을 열어둠.

### 2단계: 양쪽 서버에 8코어(CPU 800%) 부하 명령 실행

> 💡 **강사님 팁:** 최근 AWS 정책이 보수적으로 바뀌어 CPU 400%로는 분산 계산 때문에 인스턴스가 잘 안 늘어날 수 있습니다. 확실한 결과를 위해 **CPU 옵션을 8로 높여서** 실행합니다.
> 
- [터미널 1]과 [터미널 2]에 각각 아래 명령어를 입력하고 **동시에 엔터**를 칩니다.

```
# CPU 8코어 부하를 10분(600초) 동안 유지 (터미널 창을 끄면 안 됨 ❌)
stress --cpu 8 --timeout 600
```

## 3. 📊 실습 2: 실시간 인스턴스 자동 확장(Scale-out) 모니터링

- **이메일 알림 확인:** 약 5분 정도 지나면 CloudWatch가 임계값(50% 이상) 돌파를 감지하고, 아까 SNS에 등록한 이메일로 경보 메일을 보냅니다.
- **인스턴스 개수 자동 증설 확인:** `EC2 인스턴스` 목록 창으로 돌아와 우측 상단 새로고침(🔄)을 누르면, 개수가 기존 2대에서 최대 용량인 5대까지 자동으로 생성(Initializing)되는 장관을 확인할 수 있습니다.

## 4. 🧹 실습 3: 이번 단원 최종 자원 삭제 가이드 (실습 종료 후)

> 🚨 **매우 중요:** 부하 테스트 완료를 확인한 후, 클라우드 비용 폭탄을 막기 위해 반드시 **정해진 순서대로** 자원을 삭제해야 합니다. 오토 스케일링 그룹을 먼저 지우지 않으면 인스턴스를 수동으로 지워도 계속 새로 부활합니다!
> 
- [ ]  **1. 오토 스케일링 그룹 삭제:** `오토 스케일링 그룹` ➡️ `web-asg` 선택 ➡️ **[삭제]** (이 작업이 완료되면 ASG가 만든 인스턴스들이 알아서 Terminated(종료) 처리됩니다.)
- [ ]  **2. 시작 템플릿 삭제:** `시작 템플릿` ➡️ 생성한 템플릿 선택 ➡️ `작업` ➡️ **[템플릿 삭제]**
- [ ]  **3. 로드 밸런서 삭제:** `로드 밸런서` ➡️ `my-alb` 선택 ➡️ **[삭제]**
